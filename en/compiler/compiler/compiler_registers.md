# Compiler — Register Management & Liveness

The XCX compile pipeline uses a dense register-based architecture. Locals are initially allocated linearly by `FunctionCompiler`, and are subsequently compressed into a minimal strictly-bounded window by the `RegisterManager`.

---

## Module Layout

```
├── liveness.rs          — LivenessAnalysis (last-use and loop-bounding checks)

```

## `LivenessAnalysis`

```rust
pub struct LivenessAnalysis {
    pub last_uses: HashMap<usize, Vec<u8>>,
    pub virtual_last_use: HashMap<u8, usize>,
}
```

Performs a backward pass over a generated bytecode array to calculate the exact instruction IP where each register's lifetime ends. This allows the JIT compiler to know when locals can be safely skipped or overwritten without `Arc` ref-counting collisions.

### Two-Pass Execution

1. **Loop Detection**: Identifies loop back-edges (`Jump`, `LoopNext`, `ArrayLoopNext`, etc.) and records their `(start_ip, end_ip)` bounds.
2. **Backward Scan**: Iterates backward from `bytecode.len() - 1` to `0`. For each opcode:
   - Call `collect_registers(op)` to extract all source and destination registers.
   - For every register seen for the first time in the backward scan, mark its `last_use` at the current `ip`.
   - **Loop Invariant Extension**: If the current `ip` lies within a loop block, the register's lifetime is artificially extended to the loop's `end_ip`. If a register is used inside a loop, it must be kept alive until the loop terminates.

`collect_registers` contains exhaustive matching to pull all `dst`, `src1`, `src2`, and implicit argument ranges (`Call`, `MethodCall`, `ArrayInit`) out of every opcode variant.

## `RegisterManager`

```rust
pub struct RegisterManager;
```

A post-compilation pass applied to the bytecode before returning the final `Chunk`. Its sole purpose is to compress the sparse virtual registers generated by `FunctionCompiler::push_reg()` into a dense, gapless block.

### `compress_registers`

```rust
pub fn compress_registers(bytecode: &mut [OpCode], num_params: usize) -> usize
```

Operates in 4 steps:

1. **Parameter Pinning**: 
   The lowest `0..num_params` registers are locked 1:1. This preserves the executor's ABI calling convention, where arguments are placed directly into the bottom of the function's register slice.
2. **Collect Used Registers**: 
   Uses `LivenessAnalysis::collect_registers` to build a `BTreeSet<u8>` of all referenced registers across the entire function block.
3. **Generate Mapping**: 
   Iterates through the sorted set of used registers. If a register is not already pinned, it is assigned the next contiguous available slot starting from `num_params`. This guarantees that the final frame size is exactly the number of unique active registers.
4. **Apply Mapping**: 
   Iterates over the `bytecode` array, mutating all registers in-place according to the mapping via exhaustive pattern matching (`apply_mapping`).

The function returns `max_slot + 1`, representing the exact number of register slots the executor must allocate for this chunk's stack frame.

## Variable Allocation Rationale

XCX chooses simple sequential allocation backed by explicit `Move` operations or temporary variable slots over complex graph-coloring allocators. This simplifies the bytecode generation phase at the cost of slightly higher register pressure. The `RegisterManager` ensures that although IDs strictly increment during compilation (often leaving gaps when blocks exit), the runtime stack allocation is never larger than the maximum number of simultaneously live virtual registers.
