# Архитектура компилятора XCX — v3.1

Компилятор XCX реализован на Rust и следует многостадийной конвейерной архитектуре.

## Конвейер компиляции

```
Исходный код
    │
    ▼
1. Лексер (Scanner)        — src/lexer/scanner.rs
    │  Результат: поток токенов
    ▼
2. Парсер (Pratt)           — src/parser/pratt.rs
    │  Результат: сырой AST (Program)
    ▼
3. Expander                — src/parser/expander.rs
    │  Результат: развёрнутый AST (include, префиксы псевдонимов)
    ▼
4. Проверка типов (Sema)   — src/sema/checker.rs
    │  Результат: проверенный, аннотированный AST
    ▼
5. Компилятор (Backend)    — src/backend/mod.rs
    │  Результат: FunctionChunk (main) + константы + функции
    ▼
6. VM                      — src/backend/vm.rs
    │  Выполнение регистрового байткода
    │  Горячие циклы → запись трасс
    ▼
7. JIT (Cranelift)         — src/backend/jit.rs
       Компиляция трасс в машинный код
```

> **Примечание**: Expander входит в `src/parser/`, но выполняется отдельной фазой после разбора, до семантики. JIT — опциональное ускорение для горячих циклов; без скомпилированной трассы выполнение байткода продолжается.

## Структура проекта

```
src/
├── lexer/
│   ├── scanner.rs      # Сканер на уровне байтов (&[u8])
│   └── token.rs        # TokenKind и Span
├── parser/
│   ├── pratt.rs        # Парсер Pratt (токены → AST)
│   ├── expander.rs     # include и префиксы псевдонимов
│   └── ast.rs          # Узлы AST (Expr, Stmt, Type, Program)
├── sema/
│   ├── checker.rs      # Проверка типов и разрешение переменных
│   ├── symbol_table.rs # Иерархическая таблица символов
│   └── interner.rs     # Интернирование строк (str → StringId)
├── backend/
│   ├── mod.rs          # Компилятор байткода (AST → OpCode)
│   ├── vm.rs           # Регистровая VM, NaN-boxing, хуки JIT
│   ├── jit.rs          # Трассовый компилятор Cranelift
│   └── repl.rs         # Интерактивный REPL
└── diagnostic/
    └── report.rs       # Отчёты об ошибках с подсветкой
```

## Система диагностики

Компилятор использует структуру `Reporter` для контекстных сообщений. Каждая ошибка включает:
- **Уровень**: ERROR или вариант HALT
- **Место**: строка и столбец
- **Подсветка**: строка исходника с подчёркиванием `~~~`

Семантические ошибки (`TypeError`) собираются в `Vec` и выводятся разом до генерации байткода. При наличии ошибок компиляция останавливается.

Ошибки VM содержат позицию из таблицы `spans` у каждого `FunctionChunk`.

## Ключевые проектные решения

### NaN-boxing значений
Все значения — один `u64`. Старшие биты тихого NaN IEEE 754 — теги типа, младшие 48 бит — полезная нагрузка. Вещественные хранятся как есть. Каждое `Value` — 8 байт без выделения кучи для скаляров.

### Регистровая VM
VM **регистровая**, не стековая. У каждого кадра — плоский `Vec<Value>` по номерам слотов. Опкоды ссылаются на регистры (`dst`, `src1`, `src2`). Временные регистры — простой bump-аллокатор (`push_reg()` / `pop_reg()`).

### Трассовый JIT (Cranelift)
Счётчик `hot_counts` на кадр. После 50 итераций по обратному ребру цикла начинается запись `TraceOp`. Готовая `Trace` компилируется Cranelift и кэшируется. Стражи (`GuardInt`, `GuardFloat`, …) при сбое возвращают в интерпретатор.

### Интернирование строк
Идентификаторы и литералы → `StringId` (u32) через `Interner`.

### Дедупликация констант
`string_constants: HashMap<String, usize>` — каждая уникальная строка один раз в таблице констант.

### Диспетчеризация методов через enum
Встроенные вызовы → `OpCode::MethodCall { kind: MethodKind, ... }`. Динамические имена — `MethodCallCustom`.

### Двухпроходная компиляция
Первый проход (`register_globals_recursive`) назначает индексы глобалам, функциям и файберам до эмиссии байткода.

### Байткод с Span
Каждый опкод сопоставлен `Span` из AST. VM выдаёт точные сообщения об ошибках по строкам.

### Файбер как корутина
Не потоки ОС. `FiberState` хранит `ip` и `locals`, возобновление кооперативное. При yield locals возвращаются в `FiberState`.

### Потокобезопасные значения
Общие коллекции — `Arc<RwLock<T>>`. Байткод и spans в `Arc<Vec<...>>` для HTTP-воркеров. `Value`: `Copy` + `Send` + `Sync`.

### Корректное завершение HTTP
Глобальный `AtomicBool` (`SHUTDOWN`) по Ctrl+C. Воркеры и главный цикл опрашивают флаг и завершаются чисто.

### Цепочка областей в SymbolTable
Родительские указатели вместо глубокого клонирования. Вход в функцию — O(1).

### Сканер на уровне байтов
Лексер на `&[u8]` без `Vec<char>`. Комментарии: `starts_with(b"---")`.
